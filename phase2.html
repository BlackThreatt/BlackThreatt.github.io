Critical API rules
Never call xQueueSend() from an ISR — use xQueueSendFromISR()Non-ISR versions acquire a scheduler lock — calling from ISR causes undefined behaviour
Always check return value: pdTRUE vs errQUEUE_FULLDropped data is silent. Silent data loss in embedded = hard-to-reproduce bugs
portMAX_DELAY blocks forever — only use when you can prove it won't deadlockIn most cases, use a finite timeout and handle the timeout case
Sensor data struct — use throughout the project
typedef struct {
    uint32_t timestamp_ms;   /* xTaskGetTickCount() at sample time */

    uint8_t  valid;          /* 1 if data is fresh */

Week 3 — Semaphores, mutexes, synchronisation bugs

Critical week
This is the most important week in Phase 2
Priority inversion and deadlock are the two most dangerous bugs in RTOS development. You must be able to reproduce them on demand, explain them precisely, and fix them confidently. These come up in virtually every embedded systems interview. Document everything — it becomes interview material.
binary semaphorecounting semaphore mutex vs semaphorepriority inheritance priority inversiondeadlockCoffman conditions
Semaphore vs Mutex — know this cold
Semaphore: signalling. One task/ISR gives, another waits. No ownership concept.Use for: ISR→task sync, task→task sync, counting resources (N buffers available)
Mutex: ownership. Taker must be giver. Enables priority inheritance.Use for: shared resource protection between tasks. NEVER from ISR.
Reading
K C Wang Ch. 6: mutual exclusion theory, semaphore internals, deadlock (Coffman conditions)
Amos Ch. 8–10: FreeRTOS semaphores, mutexes, priority inheritanceRead the FreeRTOS source for xSemaphoreTake — compare with xQueueReceive
Debug scenario 1 — reproduce priority inversion
Setup: 3 tasks — Low (P1), Medium (P2), High (P3). Low takes a binary semaphore and holds it while doing "work" (vTaskDelay). High tries to take the same semaphore and blocks. Medium has no semaphore involvement but preempts Low.
Observe: High is blocked on semaphore held by Low. Medium preempts Low (Medium > Low priority). High starves. This is priority inversion — Medium effectively runs at higher priority than High.
Measure: Toggle GPIO in High task when it runs. Measure on oscilloscope. High LED should be dark while Medium runs.
Fix: Replace binary semaphore with xSemaphoreCreateMutex(). FreeRTOS priority inheritance temporarily raises Low to High's priority. Medium can no longer preempt it. Re-measure — High runs within microseconds of Low releasing.
Debug scenario 2 — reproduce deadlock
Setup: 2 tasks, 2 mutexes (A and B). Task1 takes A then B. Task2 takes B then A. Both tasks run.
Observe: Both tasks block forever. CPU is idle. Watchdog (if present) resets the system.
Fix option 1: Always acquire mutexes in the same global order (A then B) in all tasks.
Fix option 2: Use xSemaphoreTake() with a timeout. If second mutex not acquired within timeout, release first and retry.
Debug scenario 3 — ISR misuse
Call xSemaphoreTake() (non-ISR version) from inside an interrupt handler. Observe HardFault or assert fire. This is one of the most common RTOS bugs in real codebases. Know why it fails and what to use instead (xSemaphoreGiveFromISR).
Project — shared log buffer
Add a shared log buffer accessed by both Aggregator and UART_Report tasks. Without mutex: observe corrupt log entries on UART. With mutex: clean entries. Demonstrate both — take a screenshot of each. Document the difference.

Deferred ISR — the production pattern
ISR (minimal): read hardware register, post to queue or give semaphore, exit. Under 1 microsecond.
Worker task: blocks on queue/semaphore, does all processing at task context priority.
This is how every well-written production RTOS driver is structured.
Task notification — faster than semaphore
/* Giving from ISR — faster than xSemaphoreGiveFromISR */
void EXTI0_IRQHandler(void) {
    BaseType_t xHigherPriority = pdFALSE;
    vTaskNotifyGiveFromISR(g_worker_task_handle, &xHigherPriority);
    portYIELD_FROM_ISR(xHigherPriority);
}

/* Taking in task — blocks until notification received */
void Worker_Task(void *pv) {
    while (1) {
        ulTaskNotifyTake(pdTRUE, portMAX_DELAY);  /* block */
        /* do deferred work here */
        Process_Sensor_Data();
    }
}

Week 5 — Memory management and DMA integration

Memory
heap schemes 1–5fragmentation static allocationDMA + semaphore double bufferingcache coherency
FreeRTOS heap schemes
heap_1: alloc only, no free. Zero fragmentation.Use when all objects allocated at startup and never freed
heap_2: free supported, best-fit. Can fragment.Avoid unless allocation sizes are all identical
heap_3: wraps malloc/free from libc.
heap_4: first-fit with coalescence. General purpose. Most common.Reduces fragmentation vs heap_2. Use this as default.
heap_5: like heap_4 but spans multiple memory regions (CCM + SRAM).
DMA + RTOS pattern
Start DMA transfer. Task blocks on semaphore.Do NOT poll for DMA complete in a task — wastes CPU
DMA complete ISR gives semaphore. Task wakes and processes buffer.
Cortex-M7 D-cache: must invalidate cache after DMA RX, clean before DMA TX.SCB_InvalidateDCache_by_Addr() / SCB_CleanDCache_by_Addr()
Static allocation — the automotive-safe approach
/* Task: statically allocated stack and TCB */
static StackType_t  s_imu_stack[256];
static StaticTask_t s_imu_tcb;

Rate-monotonic schedulability
Liu & Layland bound: for n tasks, RM schedulable if U ≤ n(2^(1/n) − 1)For large n this approaches ln(2) ≈ 69%. Above this bound: not guaranteed schedulable under RM.
U = sum of (WCET_i / Period_i) for all tasksMeasure WCET with DWT_CYCCNT — not with vTaskGetRunTimeStats (too coarse)

HardFault handler — decode the exception frame
/* ARM Cortex-M exception frame pushed to stack on fault:
   R0, R1, R2, R3, R12, LR, PC, xPSR (in that order)
   The PC value tells you exactly where the fault occurred.       */

/* Fault status registers — read immediately in handler */
#define SCB_CFSR   (*((volatile uint32_t *)0xE000ED28))  /* Combined */
#define SCB_HFSR   (*((volatile uint32_t *)0xE000ED2C))  /* HardFault */
#define SCB_MMFAR  (*((volatile uint32_t *)0xE000ED34))  /* MemManage addr */
#define SCB_BFAR   (*((volatile uint32_t *)0xE000ED38))  /* BusFault addr */

typedef struct {
    uint32_t r0, r1, r2, r3, r12;
    uint32_t lr, pc, xpsr;
} ExcFrame_t;

void HardFault_Handler_C(ExcFrame_t *frame) {
    /* Log to backup SRAM (survives reset) */
    FAULT_LOG->pc   = frame->pc;
    FAULT_LOG->lr   = frame->lr;
    FAULT_LOG->cfsr = SCB_CFSR;
    FAULT_LOG->hfsr = SCB_HFSR;
    FAULT_LOG->valid = 0xDEADBEEF;

    /* Log over UART if still functional */
    LOG_ERROR("HardFault at PC=0x%08X LR=0x%08X CFSR=0x%08X",
               frame->pc, frame->lr, SCB_CFSR);

    /* Force reset via NVIC */
    NVIC_SystemReset();
}

/* Assembly trampoline — switches to correct stack before calling C handler */
__attribute__((naked)) void HardFault_Handler(void) {
    __asm volatile (
        "tst lr, #4         \n"
        "ite eq             \n"
        "mrseq r0, msp      \n"
        "mrsne r0, psp      \n"
        "b HardFault_Handler_C \n"
    );
}
