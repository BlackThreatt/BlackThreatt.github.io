Week 1 — Professional craft and MISRA-C compliance

Standards
Why MISRA-C matters for automotive hiring
MISRA-C 2012 is mandatory in ISO 26262 (automotive functional safety) and IEC 61508. Most automotive embedded job descriptions list it explicitly. You don't need every rule memorised — you need to understand the rationale, have applied it to real code, and be able to discuss specific rules with examples. "I ran clang-tidy with MISRA checks on my FOC motor controller and resolved all mandatory violations" is a concrete differentiator.
Mandatory rulesRequired rules Advisory rulesdeviation records clang-tidycppcheckstatic analysis
Rule category structure
Mandatory (16 rules): Must never be violated. No deviation permitted under any circumstances.Examples: no undefined behaviour, no unreachable code, memory not accessed out of bounds
Required (~100 rules): Must comply unless a formal written deviation is documented with justification.Examples: all switch statements have a default clause, no implicit conversions that lose precision
Advisory (~50 rules): Best practice. Informal deviation acceptable.Examples: prefer named constants over literals, limit function complexity
High-value rules to understand deeply — these come up in interviews
Rule 10.1–10.8: essential type model — no implicit conversions that change sign or lose precisionWhy it matters: silent truncation causes a large proportion of embedded integer bugs. Know 2–3 examples.
Rule 14.3: no invariant boolean conditions (dead code)Compiler warnings miss these. Static analysis finds them. Dead conditions = logic bugs or unreachable error handling.
Rule 15.5: only one return statement per functionWhy: multiple returns complicate coverage analysis and can bypass cleanup code. Controversial but defensible.
Rule 17.2: no recursive functionsWhy: stack depth becomes unbounded. In safety-critical systems, stack overflow must be provably impossible.
Rule 21.3: no dynamic memory allocation (malloc/free) after initialisationWhy: fragmentation and non-deterministic time. You already did this in Phase 2 (static FreeRTOS allocation).
Dir 4.1: no runtime errors — no UB, overflow, divide-by-zero in any reachable pathThe most fundamental safety requirement. Your code must be provably free of UB in all reachable paths.

Interview Q&A — click to expand
What does volatile do and when exactly do you need it? ▼
volatile tells the compiler the variable's value may change outside normal program flow — so it must never be cached in a register or optimised away. Use it for: (1) hardware memory-mapped registers — the register value changes when hardware writes it, (2) variables shared between an ISR and a task/main loop — the ISR can write at any moment, (3) variables used in busy-wait loops where the loop condition would otherwise be optimised away. Common bug: task polls a flag set by an ISR without volatile — compiler caches the flag in a register and the loop never sees the ISR update. Critical point: volatile does NOT provide atomicity. A read-modify-write on a volatile variable is still not atomic on Cortex-M without a critical section. volatile and atomic are separate concerns.
Can const and volatile be combined? Give a real example. ▼
Yes — they are completely orthogonal. const means "this code will not write to the variable." volatile means "the value may change outside my control." A read-only hardware status register is volatile const: your code never writes to it (const), but its value changes as hardware updates it (volatile). Declaration: volatile const uint32_t *const STATUS_REG = (volatile const uint32_t *)0x40020000; — the register address is const (pointer doesn't move), the register contents are volatile const (we read it, hardware writes it).
What is undefined behaviour? Give three embedded-specific examples. ▼
Undefined behaviour (UB) is code whose outcome the C standard does not define — the compiler can generate anything, including code that appears to work at -O0 and silently breaks at -O2. Three embedded examples: (1) Signed integer overflow: int x = INT_MAX; x++; — UB. The compiler assumes this never happens and can delete range checks that depend on it. Use uint32_t and explicit wrapping. (2) Type-aliasing violation: casting a float* to uint32_t* and reading it — UB. Use memcpy() or a union instead. (3) Reading an uninitialised variable: uint8_t buf[64]; if (buf[0] == 0xFF) — UB. The compiler may treat the branch as dead code. The fix is always explicit initialisation. MISRA Dir 4.1 mandates that all execution paths be free of UB.
Implement an ISR-safe ring buffer in C on a whiteboard ▼
Key design decisions to state before writing code: (1) Power-of-two size so head/tail wrap with bitmask, no division. (2) head = write index (ISR writes), tail = read index (task reads) — single producer, single consumer, lock-free if indices are volatile uint32_t on Cortex-M (32-bit reads/writes are atomic). (3) Declare head and tail volatile. (4) Full condition: (head - tail) == size. Empty: head == tail. Core write: buf[head++ & mask] = item; Core read: item = buf[tail++ & mask]; If multiple writers exist, need a critical section around the write. State this explicitly. The volatile + power-of-two approach is the standard embedded lock-free ring buffer pattern.
What is the difference between #define and const for constants in embedded C? ▼
#define is a preprocessor text substitution — it has no type, no scope, cannot be passed by pointer, does not appear in the debugger symbol table. const is a typed variable with scope and visibility that does appear in debug symbols. In embedded C, const variables declared at file scope with static are typically placed in .rodata section in flash — same as #define in terms of RAM usage. Prefer typed constants (static const uint32_t, enum) over #define for better type checking, debugger visibility, and MISRA compliance (Rule 5.4 recommends macro names be distinct, and many coding standards prefer typed constants). Use #define only for conditional compilation guards and function-like macros where a typed constant cannot substitute.
Explain priority inversion. How did you reproduce and fix it? ▼
Priority inversion occurs when a high-priority task is blocked waiting for a resource held by a low-priority task, while a medium-priority task runs freely — effectively inverting the execution order of high and medium. I reproduced it deliberately in Phase 2: three tasks (Low P1, Medium P2, High P3). Low holds a binary semaphore doing "work" via vTaskDelay. High tries to take the same semaphore and blocks. Medium has no semaphore involvement but preempts Low (because Medium priority > Low). High is starved even though it outranks Medium. I measured this on an oscilloscope: High's GPIO was dark while Medium's was on. Fix: replace the binary semaphore with a FreeRTOS mutex. Mutexes implement priority inheritance — when High blocks on the mutex held by Low, the kernel temporarily raises Low's priority to match High's, so Low can finish and release. Medium can no longer preempt it. After release, Low returns to P1. I re-measured on scope and verified High ran within microseconds of Low releasing the mutex.
What is the difference between a semaphore and a mutex? ▼
A semaphore is a signalling mechanism. One entity gives it, another waits for it. There is no ownership — the task that takes it doesn't have to be the one that gives it. Use for: ISR-to-task synchronisation, task-to-task signalling, counting available resources. A mutex is a mutual exclusion mechanism with ownership. Only the task that takes the mutex may give it. This ownership enables priority inheritance — the scheduler knows which task holds the mutex and can boost its priority to prevent inversion. Because ownership is required, a mutex cannot be given from an ISR (ISRs have no task context). The rule: use semaphores for signalling, use mutexes for protecting shared data between tasks.
How would you design a safe watchdog system for a multi-task RTOS application? ▼
A naive watchdog that kicks IWDG in main() or a single task is an anti-pattern: if any other task hangs, the watchdog never knows. The correct pattern — which I implemented in Phases 2 and 3 — is per-task heartbeat tokens with a dedicated watchdog supervisor. Each task has a bitmask token. At the end of each execution cycle, it sets its bit in a shared atomic flags word (single STR on Cortex-M is atomic). A WatchdogMonitor task runs periodically and checks whether all task flags have been set since the last check. If all flags are present: kick IWDG and clear flags. If any flag is missing: that task didn't run within its period — log the failing task name to backup SRAM and do NOT kick IWDG, allowing the hardware watchdog to reset the system. On next boot, the log shows which task caused the reset. This guarantees the IWDG is only refreshed when the entire system is healthy, not just one task.
What are the FreeRTOS heap schemes and when would you use each? ▼
heap_1: allocation only, no free. Zero fragmentation. Use when all objects are allocated at startup and never freed — the simplest and safest option. heap_2: free supported, best-fit algorithm. Can fragment over time. Avoid unless all allocations are the same size. heap_3: wraps libc malloc/free — thread-safe wrapper. Depends on your C library's heap quality. heap_4: first-fit with block coalescence — adjacent free blocks are merged. Best general-purpose choice. Reduces fragmentation compared to heap_2. heap_5: like heap_4 but spans multiple non-contiguous memory regions (for example, on-chip CCM + SRAM + external SRAM on STM32H7). For automotive and safety-critical work, the safest choice is to not use dynamic allocation at runtime at all — allocate everything statically at startup using xTaskCreateStatic, xQueueCreateStatic, etc. I did this in Phase 2 and verified zero runtime heap usage.
Explain FOC and why it is better than trapezoidal commutation ▼
Trapezoidal commutation drives one phase high, one low, one floating at each of 6 rotor positions. It's simple and fast but produces torque ripple — the torque-producing current vector jumps in 60-degree steps rather than rotating smoothly, causing vibration, audible noise, and reduced efficiency especially at low speeds. Field-Oriented Control solves this by continuously rotating the current vector to stay perpendicular to the rotor flux — the condition for maximum torque per ampere. The Clarke transform converts three-phase currents to a stationary 2D frame (alpha-beta). The Park transform converts to a frame rotating synchronously with the rotor (d-q). In this frame, currents appear as DC values: Id (flux component, set to zero for maximum efficiency in a PMSM) and Iq (torque component, commanded by the outer speed loop). Two PID controllers regulate Id and Iq — giving precise, fast torque control. Outputs go through Inverse Park and SVPWM to generate three-phase PWM. Result: smooth torque at all speeds, higher efficiency, lower acoustic noise, faster dynamic response. Every automotive traction drive and EPS motor controller uses FOC for these reasons.
Why must the inner current loop be faster than the outer speed loop in cascade control? ▼
In cascade control, the outer speed loop commands a setpoint to the inner current loop. The outer loop treats the inner loop as an ideal, instantaneous process. If the inner loop responds too slowly, the outer loop issues commands that haven't been executed yet — creating a feedback delay that destabilises the outer loop. The bandwidth separation rule requires the inner loop to be at least 5–10× faster than the outer loop. In my implementation: inner current loop at 20kHz with ~2–5kHz bandwidth; outer speed loop at 1kHz with 50–200Hz bandwidth — a separation factor of 10–20, which is conservative and stable. There's also a physical reason: current must be controllable faster than speed changes, because current produces torque, and torque changes speed. The current loop bandwidth must comfortably exceed the mechanical bandwidth of the motor-load system.
What is integral windup and how do you prevent it? ▼
Integral windup occurs when the PID output is saturated (the actuator is at its limit) but the integral term continues accumulating error. When the system comes out of saturation, the integrator has grown so large it drives the output in the wrong direction, causing overshoot or oscillation before it slowly unwinds. I implemented clamping anti-windup in my motor controller: the integrator only updates when the total PID output is within the output limits, or when the output is saturated but the error and integrator have opposite signs (meaning the integrator is already working to desaturate). This is the simplest effective method — easy to implement in fixed-point, predictable behaviour, trivially verified in unit tests (I have a specific test for it). The alternative — back-calculation anti-windup — feeds the difference between saturated and unsaturated output back through a gain, giving smoother recovery but adding a tuning parameter.
What is phase margin and why does it matter for your PID tuning? ▼
Phase margin is the amount of additional phase lag the open-loop system can tolerate before the closed-loop system becomes unstable — measured at the gain crossover frequency (where |G(jω)| = 1, or 0 dB). A system with zero phase margin will oscillate continuously. The engineering target is 45–60 degrees. Less than 30 degrees gives an oscillatory, lightly-damped response that is sensitive to parameter variation. More than 80 degrees gives a sluggish overdamped response. In PID tuning: increasing Kp reduces phase margin (moves the gain crossover to a higher frequency where the plant has more phase lag). Adding derivative action adds phase (good — improves margin). Adding integral action subtracts phase at high frequencies (use with care). I computed the phase margin of my motor + PID system using the Python control library's control.margin() function as part of Week 5 of Phase 3, verified it was in the 50–60 degree range before connecting to hardware, and compared the predicted step response with the measured oscilloscope trace.
Design an embedded motor controller for an electric power steering system ▼
Start with clarifying questions: motor type (PMSM assumed), bus voltage (12V or 48V), torque/speed specs, safety integrity level (ISO 26262 ASIL B or D typical for EPS), communication bus (CAN/CAN-FD/LIN). For a 48V PMSM with ASIL B: MCU: dual-core (e.g. STM32H7 or Renesas RH850) — one core runs FOC at 20kHz, second core runs independent safety monitoring and cross-checks. FOC architecture: same cascade as my Phase 3 project — Clarke/Park in fixed-point, Id/Iq PIDs, SVPWM, outer torque loop commanded by steering ECU via CAN. Safety mechanisms: hardware overcurrent protection (comparator + gate driver ENABLE), software 3-layer protection (ISR + task + thermal derating), IWDG with per-task heartbeats, plausibility checks on torque sensor input (redundant sensors compared), cross-core comparison of safety-critical signals. Interface: CAN-FD for torque command and status. Diagnostics: DTC storage in NVM, UDS over CAN for fault readout. Testing strategy: MISRA-C compliance, MC/DC coverage targets, HIL testing of full torque control loop, FMEA for each safety mechanism. This demonstrates both technical depth and awareness of the production safety context.
How would you approach testing a safety-critical embedded system? ▼
Testing at four levels with documented coverage targets. Unit tests: host-side with mocked hardware, for all algorithmic components (PID, transforms, state machines, protocol codecs). Coverage target: MC/DC for ASIL C/D, branch coverage for ASIL A/B. Integration tests on target hardware: verify inter-module interactions — ADC triggers correct ISR, ISR posts to queue correctly, task wakes within deadline. HIL tests: real MCU, simulated motor plant (DAC-driven current feedback, simulated encoder). All fault injection scenarios tested: overcurrent, overvoltage, encoder loss, communication timeout. System tests: real motor, full operating envelope, thermal stress, power cycling, long-duration soak. Automation: CI pipeline runs unit tests on every commit, HIL tests nightly. Documentation: test plan with pass/fail criteria, coverage report, requirements-to-test traceability matrix. For ISO 26262: safety requirements → safety mechanisms → verification tests must all be traceable in both directions. My Phase 4 GitHub CI pipeline with Ceedling demonstrates I already work this way.
